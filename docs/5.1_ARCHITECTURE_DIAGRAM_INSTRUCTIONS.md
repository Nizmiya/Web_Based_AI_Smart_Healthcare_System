# 5.1 Architecture Diagram – Full Features & Functions

## Purpose
Draw a **system architecture diagram** that shows **Model Training**, **Backend**, and **Frontend** with their **full features and main functions**. The diagram should show how data and control flow from training → deployment → user.

---

## PART A – MODEL TRAINING (Offline / Jupyter)

**Place in diagram:** A separate block "Model Training Pipeline" or "ML Training (Offline)". It produces files that the Backend later loads.

### Components / Steps to Show

| Step | Function / Feature | Output |
|------|-------------------|--------|
| 1 | **Data Load** – Load CSV (e.g. diabetes, heart, kidney dataset) | Raw DataFrame |
| 2 | **EDA** – Shape, missing values, basic statistics | Insights |
| 3 | **Target Distribution** – 0/1 count, balance check | — |
| 4 | **Correlation Analysis** | — |
| 5 | **Preprocessing** – Missing value handle, encoding (e.g. label encoders for kidney), scaling (StandardScaler where needed) | Processed data |
| 6 | **Train-Test Split** – 80% train, 20% test | X_train, X_test, y_train, y_test |
| 7 | **Model Training** – Train disease model (e.g. Decision Tree, Gradient Boosting, XGBoost per disease) | Trained model |
| 8 | **Evaluation** – Accuracy, confusion matrix, ROC curve | Metrics |
| 9 | **Best Threshold** – Youden Index from ROC, get best_threshold | best_threshold |
| 10 | **Save Artifacts** – joblib.dump(model), joblib.dump(scaler), joblib.dump(encoders), json.dump(threshold) | `ml_models/` files |

### Outputs (for Backend)

- `diabetes_model.pkl`
- `heart_disease_model.pkl`, `heart_disease_scaler.pkl`
- `kidney_disease_model.pkl`, `kidney_disease_scaler.pkl`, `kidney_disease_encoders.pkl`
- `threshold.json` (keys: diabetes, heart_disease, kidney_disease → Youden threshold each)

**Connection in diagram:** Arrow from "Model Training" to "ml_models / File Store" or "Backend (loads models)".

## PART B – BACKEND (FastAPI)

**Place in diagram:** Central block "Backend (FastAPI)" with sub-modules or list of main functions/APIs.

### B.1 Modules & Endpoints (Functions)

| Module | Prefix | Main Endpoints / Functions |
|--------|--------|----------------------------|
| **Auth** | `/api/v1/auth` | POST register, POST login, GET me |
| **Predictions** | `/api/v1/predictions` | POST diabetes, POST heart-disease, POST kidney-disease, GET history, GET all, POST {id}/review |
| **Users** | `/api/v1/users` | GET profile, PUT profile, GET patients/{id}/profile, POST patients/{id}/remarks, GET all, GET patients |
| **Admin** | `/api/v1/admin` | POST doctors, GET doctors, DELETE doctors/{id}, PATCH users/{id}, POST/DELETE patients/{id}/assign-doctor, GET/PUT settings, GET reports/summary, GET models |
| **Consultations** | `/api/v1/consultations` | POST, GET list, GET {id}, PATCH {id} |
| **Notifications** | `/api/v1/notifications` | GET list, PUT {id}/read |
| **Patient Records** | `/api/v1/patient-records` | POST store, GET patient/{id}, GET my-records, PUT update/{id}, DELETE delete/{id}, GET all |
| **Chatbot** | `/api/v1/chatbot` | POST message, GET models |

### B.2 Core Backend Functions (inside Predictions / ML flow)

| Function | Purpose |
|----------|---------|
| **load_model(disease_type)** | Load `{disease}_model.pkl` from ml_models |
| **load_scaler(disease_type)** | Load `{disease}_scaler.pkl` (heart, kidney) |
| **load_encoders(disease_type)** | Load `{disease}_encoders.pkl` (kidney) |
| **_load_threshold_from_json(disease_type)** | Read threshold from threshold.json for disease |
| **get_risk_level(risk_value, disease_type, prediction)** | prediction=0 → Low; else use threshold: < threshold → Medium, < 0.8 → High, else Critical |
| **Predict flow** | Receive input → scale/encode if needed → model.predict(), model.predict_proba() → risk level → Gemini recommendation → video select → save prediction → return JSON |

### B.3 Backend Connections

- **Backend ↔ MongoDB:** All persistent data (users, predictions, notifications, patient_records, consultations, system_settings).
- **Backend ↔ ml_models/:** Read-only load of .pkl and threshold.json.
- **Backend ↔ Gemini API:** Optional call to get recommendation text (disease + risk level).

### C.2 Frontend Functions / Features

- **Auth:** Store JWT and user in localStorage; redirect by role (patient → /patient/dashboard, etc.).
- **API client:** Central `api` (auth, predictions, users, admin, consultations, notifications, patientRecords, chatbot).
- **Prediction flow:** Form submit → POST to /predictions/{disease} → display disease Yes/No, risk level (color), recommendation text, embed YouTube video.
- **Role-based access:** Middleware / route guard; patient/doctor/admin see only their routes.

### C.3 Frontend Connection

- **User (Browser) ↔ Frontend:** HTTPS, user interacts with forms and views.
- **Frontend ↔ Backend:** REST API calls (JSON), JWT in Authorization header.

## How to Draw the Full Architecture

1. **Top:** User / Browser.
2. **Below:** Frontend (Next.js) – optionally list "Patient / Doctor / Admin" and main features (predictions, history, profile, consultations, admin management).
3. **Center:** Backend (FastAPI) – show modules (Auth, Predictions, Users, Admin, Consultations, Notifications, Patient Records, Chatbot) and core functions (load_model, get_risk_level, predict).
4. **Left or bottom:** Model Training pipeline – box with steps (Data → EDA → Preprocess → Train → Evaluate → ROC → Threshold → Save) and arrow to ml_models.
5. **Right or bottom:** Data layer – MongoDB, ml_models (files), Gemini API.
6. **Arrows:**  
   - User → Frontend  
   - Frontend → Backend (REST, JWT)  
   - Backend → MongoDB  
   - Backend → ml_models (read)  
   - Backend → Gemini  
   - Model Training → ml_models (write)
