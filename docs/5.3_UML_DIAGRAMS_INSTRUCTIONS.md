# 5.3 UML Diagrams – Instructions

## Purpose

Draw **UML diagrams** for the system. Typically you need at least:

1. **Use Case Diagram** – actors and main use cases.
2. **Class Diagram** (or **Component Diagram**) – main modules/classes.
3. **Sequence Diagram** (optional) – one main flow (e.g. prediction flow).

Below are instructions for each.

---

## 1. Use Case Diagram

### Actors

- **Patient** – uses the system for predictions, profile, history.
- **Doctor** – reviews predictions, views patients, adds remarks, consultations.
- **Admin** – manages users, doctors, patients, settings, reports, ML models.

### Use Cases (by actor)

**Patient**

- Login / Register
- View Dashboard
- Take Diabetes / Heart / Kidney Prediction
- View Prediction History
- View/Update Profile (including medical history, allergies, medications)
- View Recommendations and Video
- View Notifications

**Doctor**

- Login
- View Dashboard
- View All Predictions / Filter by risk
- Review Prediction (add comment, notify patient)
- View Patients List
- View Patient Profile and Add Medical Remark
- Manage Consultations (create, list, update status)
- View Notifications (high-risk alerts)

**Admin**

- Login
- View Dashboard
- Manage Patients (list, activate/deactivate, assign doctor)
- Manage Doctors (create, list, delete)
- View Reports / Analytics
- View ML Models List
- Manage System Settings
- View Alerts

### How to Draw

- **Stick figures** for Patient, Doctor, Admin.
- **Ovals** for each use case (group by actor if needed).
- **Lines** from actor to use case (association).
- **System boundary:** one big rectangle “Smart Healthcare System” containing all use cases.
- Optionally: “Include” or “Extend” between use cases (e.g. “Take Prediction” includes “View Result”).

---

## 2. Class Diagram (or Component Diagram)

### Option A – Simplified Class Diagram (main backend “concepts”)

| Class / Module    | Attributes / Responsibilities                                                   |
| ----------------- | ------------------------------------------------------------------------------- |
| **User**          | id, email, full_name, role, is_active; can be Patient, Doctor, Admin.           |
| **Prediction**    | user_id, disease_type, prediction, risk_percentage, risk_level, recommendation. |
| **Notification**  | user_id, type, title, message, is_read.                                         |
| **Consultation**  | patient_id, doctor_id, scheduled_at, status, notes.                             |
| **PatientRecord** | patient_id, disease_type, parameters.                                           |

Relationships: User 1──* Prediction; User 1──* Notification; User (patient) 1──* Consultation; User (doctor) 1──* Consultation; etc.

### Option B – Component Diagram (high level)

- **Frontend** (Next.js): Pages, API client.
- **Backend** (FastAPI): Auth, Predictions, Users, Admin, Consultations, Notifications.
- **Database**: MongoDB (users, predictions, …).
- **ML Service**: Load model, predict, risk level.
- **External**: Gemini API.

Draw boxes for each component and arrows for “uses” or “calls”.

---

## 3. Sequence Diagram – Prediction Flow

**Actors:** Patient (User), Frontend, Backend, Model, MongoDB.

**Steps:**

1. User fills form → Frontend.
2. Frontend → Backend: POST /predictions/diabetes (with token and form data).
3. Backend: validate token, load model and threshold.
4. Backend → Model: predict(input), predict_proba(input).
5. Backend: compute risk level from threshold.json.
6. Backend: (optional) call Gemini for recommendation.
7. Backend: select video by risk level.
8. Backend → Frontend: JSON response (prediction, risk_level, recommendation, video).
9. Backend → MongoDB: save prediction (optional).
10. Frontend: display result and embed video.

Draw lifelines (vertical lines) for each actor/component; horizontal arrows for messages; label with API or action name.

---

## Suggested Tool

- Draw.io (has UML shapes), PlantUML, Lucidchart, or Visio.
- For sequence diagram: Draw.io, Mermaid (in .md), or PlantUML.

## Checklist

- [ ] Use Case Diagram: 3 actors, main use cases per actor, system boundary.
- [ ] Class or Component Diagram: main entities or backend/frontend/DB components.
- [ ] (Optional) Sequence Diagram: one complete flow (e.g. prediction).
- [ ] Each diagram has a title and is exported as image/PDF for your document.
