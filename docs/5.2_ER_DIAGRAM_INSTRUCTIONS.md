# 5.2 ER Diagram – Instructions

## Purpose
Draw an **Entity–Relationship diagram** for the **database (MongoDB)** used by the backend. Even though MongoDB is NoSQL, you can model the main “entities” and their relationships like an ER diagram.

---

## Entities (Main Collections / “Tables”)

### 1. **users**
- **Attributes:** `_id`, `email`, `full_name`, `phone`, `password` (hashed), `role` (patient | doctor | admin), `is_active`, `created_at`
- **Optional (patient):** `age`, `gender`, `address`, `emergency_contact`, `medical_history` (array), `allergies` (array), `current_medications` (array), `doctor_remarks` (array of sub-docs), `assigned_doctor_id` (FK → users)
- **Optional (admin):** `created_by` (FK → users)
- **Key:** `_id` (PK). `email` unique.

### 2. **predictions**
- **Attributes:** `_id`, `user_id` (FK → users), `disease_type` (diabetes | heart_disease | kidney_disease), `prediction` (0/1), `risk_percentage`, `risk_level`, `recommendation`, `created_at`
- **Optional:** `doctor_review` (sub-doc: doctor_id, doctor_name, comment, reviewed_at), `reviewed` (boolean)
- **Key:** `_id` (PK). `user_id` references users.

### 3. **notifications**
- **Attributes:** `_id`, `user_id` (FK → users), `type`, `title`, `message`, `is_read`, `created_at`
- **Optional:** `prediction_id` (FK → predictions)
- **Key:** `_id` (PK). `user_id` references users.

### 4. **patient_records**
- **Attributes:** `_id`, `patient_id` (FK → users), `disease_type`, `parameters` (embedded key-value), `created_at`, `updated_at`
- **Key:** `_id` (PK). `patient_id` references users.

### 5. **consultations**
- **Attributes:** `_id`, `patient_id` (FK → users), `doctor_id` (FK → users), `scheduled_at`, `status` (scheduled | completed | cancelled), `notes`, `created_at`
- **Key:** `_id` (PK). `patient_id`, `doctor_id` reference users.

### 6. **system_settings**
- **Attributes:** `_id` (e.g. "global"), `systemName`, `maxFileSize`, `sessionTimeout`, `emailNotifications`, `smsNotifications`, `maintenanceMode`, `allowRegistration`
- **Key:** `_id` (PK).

---

## Relationships (edges)

| From | To | Relationship | Description |
|------|----|--------------|-------------|
| users | users | 1:1 (optional) | Patient has one `assigned_doctor_id` → users (role=doctor). |
| users | predictions | 1:N | One user has many predictions. |
| users | notifications | 1:N | One user has many notifications. |
| users | patient_records | 1:N | One patient has many records. |
| users | consultations | 1:N (as patient) | One patient has many consultations. |
| users | consultations | 1:N (as doctor) | One doctor has many consultations. |
| predictions | notifications | 1:N (optional) | A notification can reference a prediction_id. |

---

## How to Draw

1. **Rectangles:** One per entity (users, predictions, notifications, patient_records, consultations, system_settings).
2. **Inside each rectangle:** List main attributes; mark PK (e.g. _id) and important FKs (e.g. user_id, patient_id).
3. **Lines between rectangles:** Connect FK to the entity it references. Label relationship (e.g. “1 user → N predictions”).
4. **Cardinality:** Use 1, N (or 1, *) on the ends of the line.

---

## Suggested Tool

- Draw.io, Lucidchart, or any ER tool (e.g. dbdiagram.io with “table” style).
- You can use “crow’s foot” or “1 : N” notation.
